# Claude Design 運用ガイド: フェーズ2 フロントエンドレビュー手順

本書は PRJ-003_LP のフェーズ2「デザインレビュー」で、Claude Design を用いて Antigravity 実装版のフロントエンドを反復的に改善するための手順をまとめたものである。

## 0. Claude Design とは

Anthropic Labs が提供する対話型のデザイン支援ツール。Claude Opus 4.7 をベースに、左側のチャットと右側のキャンバスからなる二分割インターフェースで、自然言語の指示と要素単位のインラインコメントによりデザインを反復生成・改善できる。

### 利用可能なプラン

Pro、Max、Team プランで利用可能。Enterprise プランはデフォルトでオフ。

### 受け付ける入力

スクリーンショット、画像、既存アセット、ワイヤーフレーム、コードリポジトリ、既存の製品デザインファイル（.fig 等）、ブランドガイドラインのテキスト記述。

### 出力形式

`.zip` ダウンロード、PDF、PPTX、Canva 連携、スタンドアロン HTML、Claude Code への引き継ぎ。Figma エクスポートはローンチ時点で非対応。

## 1. 事前準備

### 1.1 プラン確認

claude.ai にログインし、ご自身のプランが Pro／Max／Team のいずれかであることを確認する。Enterprise の場合は管理者に Claude Design の有効化を依頼する。

### 1.2 デザインシステム素材の用意

Akuboxel のブランド資産を Claude Design に登録するための素材を、`projects/PRJ-003_LP/` 配下から集める。

| 素材 | 取得先 | 用途 |
|------|-------|------|
| カラーパレット | `requirements.md` 4節「デザイン方針」 | ベース白／淡グレー、テキストダークグレー、アクセント・グレイッシュブルー |
| タイポグラフィ | 同上 | 和文 Noto Sans JP、欧文 Inter、本文 16px、見出し 4 段階 |
| ブランドキャッチコピー | `content/thanks-page.md`、要件定義書 1 節 | 「VRC向けワールド・アクセサリー制作」「あなたの世界観を形に」等 |
| ロゴ | 別途用意（未作成の場合は Antigravity か Claude Design で初稿生成可） | ヘッダー・フッター用 |
| 参考サイトスクリーンショット | studio.umeugu.com の主要セクション | テイストの参照 |
| 実装版スクリーンショット | Antigravity 実装後の `https://akuboxel.com/`（または開発環境） | レビュー対象 |

### 1.3 レビュー対象 URL の準備

Antigravity がフェーズ1を完了し、`https://akuboxel.com/` または ConoHa の開発用サブドメインに初稿が公開されている状態を前提とする。本番反映前の状態でもよい。

## 2. Claude Design セッションの開始

### 2.1 Claude Design を開く

claude.ai にログイン後、左サイドバーの「パレット型アイコン」または `https://claude.ai/design` から Claude Design を起動する。

### 2.2 プロジェクトを新規作成

- 「+ 新しいプロジェクト」を選択
- プロジェクト名: `Akuboxel LP Review`
- 説明: `VRChat向け3DCGクリエイター Akuboxel のポートフォリオ兼LP（akuboxel.com）のフロントエンドレビュー`

### 2.3 デザインシステムの登録

「設定」または「デザインシステム」タブで、Akuboxel のブランド資産を登録する。

登録方法は以下のいずれかを選ぶ。

- 方法A: 自動抽出。実装版の URL（プレビュー環境）またはスクリーンショットを与え、「このサイトからデザインシステムを抽出してください」と依頼
- 方法B: 手動入力。カラー HEX、フォント名、サイズスケール、間隔ルールをチャットで指示
- 方法C: GitHub リポジトリ連携。`assets/css/main.css` の CSS カスタムプロパティを読み込ませる

推奨は方法A＋方法Bの併用。自動抽出した結果を確認し、要件定義書 4節「デザイン方針」と乖離している項目を手動で補正する。

## 3. 初回レビュー

### 3.1 セクション別にスクリーンショットを取得

実装版の各セクションを、デスクトップ（1440px）とモバイル（375px）で撮影する。Chrome DevTools のレスポンシブモードを推奨。

撮影対象セクション

1. ヒーロー
2. Works（制作実績カード）
3. Services（対応業務）
4. Pricing（ワールド制作パッケージ）
5. Menu（単品メニュー）
6. Discount Plan（割引プラン）
7. Flow（制作フロー）
8. About（自己紹介）
9. Channels（依頼チャネル）
10. FAQ
11. 問い合わせフォーム
12. フッター

### 3.2 初回プロンプトの送信

チャット欄に、目標・レイアウト・コンテンツ・オーディエンスを含めて指示する。

例文。

```
このサイトは VRChat 向け 3DCG クリエイター Akuboxel（運営: 大山 圭一朗）の
ポートフォリオ兼ランディングページです。
以下のセクションのスクリーンショットを順に共有しますので、
ミニマル・ニュートラルなテイスト（白／淡いグレー基調、アクセントは
グレイッシュブルー1色）を維持しつつ、視覚的バランス、タイポグラフィ
スケール、余白の運用、モバイル時の破綻有無を観点にレビューしてください。

オーディエンスは VRChat 個人ユーザー、個人イベント主催者、
VTuber・配信者個人です。発注の判断材料となる料金や作例が
明確に伝わることを優先してください。

最初に確認するセクション: ヒーロー
（スクリーンショットを添付）
```

### 3.3 出力の受領

Claude Design は、キャンバス上にリファイン案を表示する。気になる箇所はコメントで質問・指示できる。

## 4. 反復改善

### 4.1 全体的な変更はチャットで

例

- 「カラースキームを少し暗めに、コントラストを上げてください」
- 「ヒーローのキャッチコピーをもう少し小さく、サブテキストとの差を 1.5 倍まで広げてください」
- 「Pricing と Menu のカードを、UGUISU のような縦長カード形式に変えてください」

### 4.2 特定要素の変更はインラインコメントで

キャンバス上の対象要素をクリックし、コメントを書く。

例

- 「このボタンのパディングを上下 +8px してください」
- 「この見出しのフォントウェイトを 600 にしてください」
- 「このカードの角丸を 16px に統一してください」

### 4.3 反復の進め方

1セッションあたり 1〜2 セクションに絞る運用を推奨する。Claude Design は反復前提のツールで、初回出力をベースにコメント追加で磨いていく方が品質が出る。

セクションごとに次のループを回す。

1. 現状スクリーンショット投入
2. レビュー依頼
3. 修正案受領
4. インラインコメントで微調整
5. 確定版を取得

## 5. 修正の取り出しと適用

### 5.1 出力形式の選択

- スクリーンショット: 修正後イメージを画像として保存（軽量、確認用）
- スタンドアロン HTML: `.zip` ダウンロードで HTML/CSS を取得（差分参照用）
- Claude Code 引き継ぎ: 修正をコード変更として直接適用させる場合に選択

### 5.2 本案件における推奨フロー

本案件は手書き HTML/CSS/JS + PHP の構成のため、Claude Design が生成したスタンドアロン HTML をそのまま採用しない。理由は次の3点。

1. PHP インクルード構造（ヘッダー・フッターの共通化）が失われる
2. `/api/contact.php` フォーム連携、`.htaccess` 等の本案件固有実装と整合しない
3. Claude Design の出力は単一 HTML ファイル前提で、`assets/css/main.css` と `assets/js/main.js` への分割が逆方向

代わりに次の流れを採る。

1. Claude Design でスクリーンショット出力を取得
2. 採用する修正項目を `notes/YYYY-MM-DD-claude-design-review.md` に箇条書きで記録
3. CSS 変数の値変更や class 追加など、人間が `assets/css/main.css` および各 HTML ファイルへ手動適用
4. 大きな構造変更（セクション組み替え等）は、Antigravity に再委譲して Claude Code 経由で適用

### 5.3 ローカル確認とデプロイ

1. 修正適用後、ローカルで Playwright スクリーンショットを再取得し、Claude Design 出力との一致を確認
2. PR 作成、CI のテストとビルドを通過
3. main へマージ → GitHub Actions が自動デプロイ
4. 本番（akuboxel.com）で目視確認

## 6. レビュー記録のテンプレート

各レビューセッションの結果を `notes/` に保存する。テンプレートを以下に示す。

```markdown
## Claude Design レビュー記録（YYYY-MM-DD）

### セッション概要
- 対象セクション: ヒーロー、Works
- レビュー時刻: HH:MM〜HH:MM
- レビュー対象URL: https://akuboxel.com/
- スクリーンショット数: 8 枚（デスクトップ4／モバイル4）

### 採用した修正
1. ヒーローのキャッチコピー: フォントサイズ 56px → 48px、ウェイト 700 → 600
2. Works カード: 角丸 8px → 16px、影 box-shadow を弱める

### 不採用の修正
1. アクセントカラーの変更案（グレイッシュブルー → 紺青）
   - 理由: 要件定義 4節「デザイン方針」のトーンから外れる

### 申し送り
- 次回セッションで Pricing と Menu のカードレイアウトを Claude Design に委ね、3案を比較する

### 適用ファイル
- `assets/css/main.css`（カスタムプロパティ `--hero-size`、`--card-radius` を更新）
- `index.html`（ヒーロー見出しのクラス変更）
```

## 7. 既知の制約と対策

### 7.1 Claude Design 側の既知の問題

- インラインコメントの永続性に問題がある場合がある。重要なコメントは本書 6 節のテンプレートに別途記録する
- コンパクトビュー保存エラーが発生することがある。こまめに `.zip` エクスポートしてバックアップする
- 大規模コードベースでラグが発生する。本案件は LP 単体なので影響は小さいが、Works 件数が増えた場合はセクション単位で分割してレビューする

### 7.2 プラン消費の注意

Claude Design は反復回数を重ねるとプロ枠の利用量を消費する。1 セッションあたりの目標を「1〜2 セクション、修正案 5 件以内」に絞り、無計画な再生成を避ける。

### 7.3 自動化されない作業

- WCAG 2.1 AA のコントラスト計測は別途 axe-core 等で確認する
- Lighthouse Performance スコアは Claude Design では検証されない
- フォーム送信、reCAPTCHA、SMTP 送信などの動的挙動は対象外

## 8. クイックリファレンス

| 操作 | 場所 | 用途 |
|------|------|------|
| 全体的な変更 | チャット欄 | カラースキーム、レイアウト構造、トーン変更 |
| 特定要素の変更 | キャンバス上クリック → インラインコメント | パディング、フォントサイズ、色、テキスト変更 |
| 比較案生成 | チャット欄で「3 つのバリエーションを並べて」 | A/B 比較、複数案検討 |
| エクスポート | 右上のエクスポートメニュー | `.zip`、PDF、PPTX、Canva、Claude Code |
| デザインシステム再登録 | 設定 → デザインシステム | ブランド資産の更新 |
