# PR作成・コードレビュー

実装完了後のチーム共有と品質保証を担う。Copilot Pro と Claude の二段構えで、要約と観点レビューを行う。

## トリガー

- 「PRを作って」「プルリクまとめて」「コミットメッセージ作って」
- 「コードレビューして」「差分を見て」
- 実装フェーズ完了の報告を受けた直後

## 役割分担

| 役割 | 担当 | 目的 |
|------|------|------|
| 差分の要約 | Copilot Pro | 変更ファイル・変更行数の機械的サマリー |
| 観点レビュー | Claude（本プロトコル） | 設計意図・セキュリティ・テストの妥当性 |
| コミットメッセージ生成 | Copilot Pro | Conventional Commits 形式の提案 |
| PR本文生成 | 本プロトコル | 背景・変更点・動作確認・レビュー依頼点 |

## 実行手順

### 1. 事前読み込み

- `projects/PRJ-XXX/requirements.md`（何を満たすべきか）
- `projects/PRJ-XXX/report.md`（Antigravity の実装報告）
- `.github/copilot-instructions.md`（コーディング規則）

### 2. 差分の把握

ユーザーから差分（git diff または変更ファイル一覧）の提示を受ける。差分が提示されない場合、ユーザーに `git diff --stat` の出力を依頼する。

### 3. 観点レビュー

以下 5 観点でチェックし、該当箇所があれば具体的な行・ファイル名を添えて指摘する。ゲーム・3D 案件では 6 つめの観点「アセット整合性」を加える。

| 観点 | 確認内容 |
|------|---------|
| セキュリティ | シークレットのハードコード、IDOR、入力検証、エラー情報の漏洩 |
| 堅牢性 | 異常系ハンドリング、冪等性、トランザクション境界 |
| 要件適合 | requirements.md の機能要件を満たしているか |
| テスト | 正常系＋境界値＋異常系の網羅、モックの乱用がないか |
| 可読性 | 命名、責務の分離、マジックナンバー、コメントの妥当性 |
| アセット整合性（ゲーム・3D案件のみ） | シーン内の参照切れ、Missing Script、LFSトラッキング漏れ、Import Settings の整合 |

### 3b. ゲーム・3D案件のバイナリ差分ルール

バイナリ資産（`.fbx`, `.png`, `.wav`, `.blend`, `.asset` など）は `git diff` で目視不能なため、以下の情報を PR 本文に含めることを必須とする。

- 追加・更新・削除したファイル名とパス
- LFS 対象かどうか（`git lfs ls-files` の出力を引用）
- 変更前・変更後のスクリーンショット（モデル・シーン・UI の変更時）
- Unity の場合、`.meta` ファイルも同時にコミットされているか

Missing Script や参照切れは、差分レビュー時点では検出できない。Unity Editor で当該シーンを開いて Console に警告が出ないかを確認する手順を、PR の「動作確認」欄に記載する。

### 4. PR本文の生成

以下のテンプレートで出力する。

```md
## 概要
[1〜2文で、この PR が何を達成するか]

## 背景
[なぜ必要か。requirements.md の該当セクション番号を引用]

## 変更点
- [ファイル or モジュール単位の変更]
- [ファイル or モジュール単位の変更]

## 動作確認
- [ ] [確認手順1]
- [ ] [確認手順2]

## レビュー依頼点
- [判断が分かれる箇所、代替案がある箇所を明示]

## 関連
- Closes #[Issue番号]
- requirements: projects/PRJ-XXX/requirements.md
```

### 5. コミットメッセージの提案

Conventional Commits 形式で 1 行目 50 文字以内、本文 72 文字折り返しを目安に提示する。

```
<type>(<scope>): <subject>

<body>

<footer>
```

type の例は `feat`, `fix`, `docs`, `refactor`, `test`, `chore`。

### 6. 保存

レビュー結果を `projects/PRJ-XXX/notes/YYYY-MM-DD-pr-review.md` に保存する。指摘事項のうち未解決のものは、同プロジェクトの README に TODO として追記する。

## 禁止事項

- 差分を見ずに「良さそうです」と回答する
- セキュリティ観点を飛ばして機能面だけ評価する
- 推測で「多分動く」と書く（動くかは実行で検証する）

## 品質チェック

提出前に以下を自問する。

- 5 観点すべてに言及したか
- 指摘は行番号・ファイル名で特定できるか
- レビュー依頼点は判断が分かれる箇所に絞れているか
